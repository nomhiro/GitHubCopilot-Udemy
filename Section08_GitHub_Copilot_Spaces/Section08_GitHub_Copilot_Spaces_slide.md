# ナレッジベースとCopilot Spaces

---

# ナレッジベースとは

## 組織固有の知識をCopilotに提供する仕組み

- **定義**: 組織固有の情報やドキュメントをCopilotに提供し、よりコンテキストに関連した応答を生成する仕組み
- **対象**: GitHub Copilot Enterprise限定の機能
- **目的**: 組織のプライベートコードベースやドキュメントを基に、パーソナライズされた提案を得る

<span class="red">試験ポイント: ナレッジベースはEnterprise限定機能です</span>

---

# ナレッジベースの主な利点

## 組織固有のコンテキストで応答品質を向上

| メリット | 説明 |
|----------|------|
| **規約準拠** | 組織のコーディング規約に沿った提案を生成 |
| **内部参照** | 内部ドキュメントを参照した回答を提供 |
| **コード理解** | プライベートコードベースのパターンを理解 |
| **一貫性** | 組織全体で一貫した応答品質を維持 |

---

# ナレッジベースの活用シーン

## どのような場面で役立つか

| 活用シーン | 説明 |
|------------|------|
| **新入社員のオンボーディング** | 社内標準やコーディング規約を素早く学習 |
| **社内標準の確認** | 内部ドキュメントに基づいた正確な回答 |
| **既存コードパターンの参照** | 組織固有の実装パターンを提案に反映 |

> ナレッジベースにより、一般的なAI回答ではなく、組織に特化した回答が得られます

---

# Copilot Spacesとの関係

## ナレッジベースを発展させた仕組み

- **Copilot Spaces**: 関連コンテンツを整理・集中化し、Copilotの応答をグラウンディング（根拠づけ）する仕組み
- ナレッジベースの概念を発展させ、より柔軟なコンテキスト管理を提供

## 所有権の種類

| 所有者 | 特徴 |
|--------|------|
| **個人** | 自分専用のスペース |
| **組織** | GitHubの権限モデルで共有可能 |

---

# スペースに追加できるソースの種類

## コンテキストソース一覧

| ソースタイプ | 説明 | 例 |
|-------------|------|-----|
| **ファイルとリポジトリ** | 特定のファイル、フォルダ、またはリポジトリ全体 | 社内ライブラリ、共通コンポーネント |
| **リンク** | PRやIssueなどのGitHubコンテンツへのURL | 関連するPR、重要なIssue |
| **アップロードファイル** | 画像、テキスト、ドキュメント、スプレッドシート | 設計書、仕様書 |
| **テキストコンテンツ** | トランスクリプトやメモなどのフリーテキスト | 会議メモ、技術メモ |

# GitHub Copilot Spacesとは

---

# GitHub Copilot Spacesの概要

## 指定されたコンテキストで一貫した回答を得る仕組み

- **特徴**: 一般的なCopilotチャットとは異なり、特定のファイル、Issue、PRにモデルを集中させる
- **メリット**: より一貫性のある再現可能な回答を実現

## 一般的なチャットとの違い

| 比較項目 | 一般チャット | Spaces |
|----------|-------------|--------|
| コンテキスト | 広範囲 | 厳密に指定 |
| 回答の一貫性 | 変動あり | 高い再現性 |
| 適用場面 | 汎用的な質問 | 特定ドメインの深い理解 |

---

# Spacesを使用すべき場面

## どのような場面で効果的か

| 使用場面 | 説明 |
|----------|------|
| **一貫した回答が必要** | 特定のサービス、Runbook、プレイブックに関する質問 |
| **予測可能な応答が必要** | 既知のデータセットについての根拠のある回答 |
| **深さを重視** | 幅広い知識より、特定分野の深い理解が必要 |

## Spacesが向かない場面

- 多数のリポジトリで広範な検出が必要な場合
- 特定のタスクやドメインを考慮していない場合

---

# スペースの作成手順

## 5ステップで作成

1. https://github.com/copilot/spaces に移動し、「スペースの作成」をクリック
2. スペースに名前を付ける
3. 所有者を選択（個人または組織）
4. 説明を追加（オプション）
5. 「保存」をクリック

## 編集について

スペースの右上隅にある「編集」をクリックすると、名前と説明をいつでも変更可能

<span class="red">試験ポイント: 説明はCopilotの回答には影響しません（人間の読者向け）</span>

---

# 所有権の選択

## 個人所有と組織所有

| 所有権タイプ | 説明 |
|------------|------|
| **個人所有** | 自分だけが使用するスペース |
| **組織所有** | GitHubの組み込みアクセス許可モデルを使用して、組織内のメンバーと共有可能 |

## 組織所有のメリット

- チームメンバー間でナレッジを共有
- GitHubの既存の権限モデルを活用
- 一貫した組織標準の維持

---

# コンテキストの2種類

## 手順（Instructions）と添付ファイル（Attachments）

| 種類 | 説明 |
|------|------|
| **手順（Instructions）** | Copilotが注目すべき内容を説明するフリーテキスト。専門分野、支援すべきタスク、避けるべきタスクを記載 |
| **添付ファイル（Attachments）** | 質問に対してより関連性の高い回答を提供するために使用 |

<span class="red">試験ポイント: スペースは常にリポジトリのメインブランチ上の最新バージョンのコードを参照します</span>

---

# 添付ファイルの種類

## 4種類のコンテンツを添付可能

| 種類 | 説明 | 含まれるもの |
|------|------|--------------|
| **ファイル/フォルダー** | GitHubリポジトリからファイルとフォルダーを添付 | コードファイル、ドキュメント、構成ファイル |
| **PR/Issue** | GitHubの問題とPRのURLを貼り付け | Issue、Pull Request |
| **ファイルアップロード** | ローカルから直接ファイルをアップロード | 画像、テキストファイル、スプレッドシート |
| **テキストコンテンツ** | フリーテキストコンテンツを入力 | トランスクリプト、ノート |

# 共有、検出可能性、ガバナンス

---

# 可視性と共有

## スペースの公開範囲を設定

| 設定項目 | 推奨事項 |
|----------|----------|
| **可視性** | 用途に応じて設定（個人/組織） |
| **タイトル** | 明確で目的に基づくタイトルを使用 |
| **説明** | 範囲、対象ユーザー、予想出力を1〜2文で記述 |

## 重要な原則

<span class="red">「スペースごとに1つのタスク」の原則を維持しましょう</span>

---

# セキュリティとアクセス

## GitHubの既存権限を継承

| ポイント | 説明 |
|----------|------|
| **権限継承** | GitHubの既存のアクセス許可に従う |
| **新規アクセス禁止** | スペースは新しいアクセスを許可しない |
| **閲覧範囲** | 閲覧者が既に閲覧権を持つコンテンツのみ表示 |

<span class="red">試験ポイント: 機密データをフリーテキストノートに貼り付けないでください</span>

---

# バージョン管理と鮮度

## コンテンツの最新性を維持

| 項目 | 動作 |
|------|------|
| **リンクされたファイル** | リポジトリの既定のブランチを反映 |
| **添付されたIssueとPR** | 変更に合わせて進化 |

## 注意点

- スペースは常にメインブランチの最新バージョンを参照
- ブランチ固有のガイダンスが必要な場合は、テキストファイルを添付

---

# チェックリスト：名前付けと所有権

## スペース作成時の確認事項

**名前付けと目的**
- 明確な目的に基づくタイトルを選択（例：「ServiceName - Onboarding Helper」）
- 「スペースごとに1つのジョブ」の原則を維持
- 範囲、対象ユーザー、予想出力を示す1〜2文の説明を記述

**所有権と可視性**
- 適切な所有者を設定（個人または組織）
- 適切な可視性を選択
- GitHubアクセス許可の継承を確認

---

# チェックリスト：セキュリティと検出可能性

## 安全で見つけやすいスペースを作る

**セキュリティとプライバシー**
- 機密データをフリーテキストに貼り付けない
- バージョン管理されたファイルへのリンクを優先
- すべての添付ソースが選択した可視性に適していることを確認

**検出可能性とドキュメント**
- スペース間で一貫した名前付け規則を使用
- 検索に役立つタグ/キーワードを説明に追加
- 組織のディレクトリ内でスペースをカタログ化

---

# チェックリスト：ガバナンス

## 継続的なメンテナンス

**ケイデンスとガバナンスの確認**
- 更新を担当する保守管理者/所有者を割り当て
- レビュー周期を設定（毎月、リリースごとなど）
- 各レビューで実施:
  - リンクの検証
  - 代表的なプロンプトのテスト
  - 例の更新
  - ノイズの多いソースの排除
- フィードバックと改善要求を追跡

# スペースで作業する際のベストプラクティス

---

# ガイドライン概要

## 6つの重要なガイドライン

| ガイドライン | 説明 |
|------------|------|
| **範囲を限定** | 質問は添付したソースに厳密に設定 |
| **@メンション禁止** | 通知されず、拡張機能も呼び出せない |
| **単一タスク集中** | 1つのタスクまたはドメインに集中 |
| **実行可能な出力** | 具体的な制約を追加し、実行可能なコードを要求 |
| **サイズ制限注意** | 警告時はソースを減らすか分割 |
| **機密データ保護** | フリーテキストに貼り付けず、リンクを優先 |

---

# Do's（すべきこと）

## 効果的にスペースを活用するために

| ガイドライン | 詳細 |
|------------|------|
| **質問の範囲を限定** | 添付したソースに厳密に設定し、回答を根拠のあるものに保つ |
| **単一タスクに集中** | 1つのタスクまたはドメインに集中させ、独自の用語を再利用 |
| **実行可能な出力を目指す** | 意図を確認→具体的な制約で絞り込む→実行可能なコードを要求 |
| **応答がずれた場合は反復** | 指示を絞り、1〜3の高品質な例を追加し、ノイズの多いソースを削除 |
| **コンテキストを新鮮に保つ** | バージョン管理ファイルをリンク、重要なソースを先頭に配置 |

---

# Don'ts（すべきでないこと）

## 避けるべき行動

| ガイドライン | 詳細 |
|------------|------|
| **@メンション禁止** | ユーザーや拡張機能を@メンションしない（通知されず、呼び出せない） |
| **外部コンテンツを期待しない** | 含まれていないコンテンツの取得を期待しない |
| **スペースの肥大化を避ける** | 単一のジョブを超えたり、コンテキスト制限を超えない |
| **機密データの貼り付け禁止** | フリーテキストノートに貼り付けず、リポジトリ内ファイルへリンク |

<span class="red">試験ポイント: @メンションでは誰にも通知されず、拡張機能も呼び出せません</span>

---

# 応答品質の改善方法

## 回答がずれた場合の対処法

1. **指示を絞る**: より具体的な指示に変更
2. **例を追加**: 1〜3の高品質な「良い出力」例を追加
3. **ソースを削除**: ノイズの多いまたは無関係なソースを削除

## サイズ警告が出た場合

- ソースを減らす
- 小さい単一ジョブのスペースに分割
- 精度と予測可能性を回復

# 試験対策ポイント

## 重要な試験ポイント

| ポイント | 内容 |
|----------|------|
| **ナレッジベース** | Enterprise限定機能 |
| **説明フィールド** | Copilotの回答には影響しない（人間向け） |
| **@メンション** | 通知されず、拡張機能も呼び出せない |
| **権限継承** | GitHubの既存権限を継承、新しいアクセスは許可しない |
| **ブランチ** | 常にメインブランチの最新バージョンを参照 |
| **機密データ** | フリーテキストに貼り付けず、リンクを優先 |
| **サイズ警告時** | ソースを減らすか、小さいスペースに分割 |

# よくある誤答パターン

## 試験で注意すべき点

| 誤答パターン | 正しい理解 |
|--------------|------------|
| 「スペースが新しいアクセスを許可する」 | GitHubの既存権限を継承するのみ |
| 「説明がCopilotの回答に影響する」 | 説明は人間の読者向け |
| 「任意のブランチを固定できる」 | 常にメインブランチを参照 |
| 「@メンションで拡張機能を呼び出せる」 | スペース内では無効 |
| 「広範な検出が必要な場合にSpacesを使う」 | 幅より深さを重視する場合に使用 |
