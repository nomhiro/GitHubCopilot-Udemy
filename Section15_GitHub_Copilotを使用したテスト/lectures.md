## レクチャー1: テスト生成の概要（ユニット・統合・その他）
**時間目安: 10分**

### GitHub Copilot Chatによる単体テストの提案

GitHub Copilot Chatは、指定したコードに基づいて単体テストを提案するために使用できます。

**主な機能:**
- メソッドのシグネチャと本文から、入力パラメーターと期待される出力値をカバーするテストケースを提案
- 既存のテストケースがある場合、追加のテストケース用のコード補完候補を生成
- コード行補完を使用して単体テストを生成し、反復的なプロセスを削減

### Copilotがテストで支援できる5つのタスク

| タスク | 説明 |
|-------|------|
| **テストフレームワークを設定する** | プロジェクトと言語に適したテストフレームワークとVS Code拡張機能の構成に関するヘルプ |
| **テストコードを生成する** | 単体テスト、統合テスト、エンドツーエンドのテストを作成 |
| **エッジケースの処理** | エッジケースとエラー状態をカバーする包括的なテストスイートを生成 |
| **失敗したテストを修正する** | テストの失敗を修正するための提案を受け取る |
| **一貫性を維持する** | プロジェクトのコーディングプラクティスに従うテストを生成するようCopilotをカスタマイズ |

### 重要な注意事項

> **重要**: 生成されたテストケースは、考えられるすべてのシナリオをカバーしているとは限りません。コードの品質を確保するには、**手動テストとコードレビュー**が必要です。

## レクチャー2: 【ハンズオン】ユニットテストの自動生成
**時間目安: 15分**

### テストフレームワークのセットアップ

テストワークフローを高速化するために、CopilotはプロジェクトのテストフレームワークとVS Code拡張機能を設定するのに役立ちます。

**セットアップ手順:**
1. チャットビューを開く
2. チャット入力フィールドに `/setupTests` コマンドを入力
3. GitHub Copilotのガイダンスに従ってプロジェクトを構成

### 単体テストを生成する4つの方法

GitHub Copilotには、単体テストを生成するための複数のツールがあります。

#### 1. チャットビュー
[チャット]ビューを使用すると、**Ask**、**Edit**、または**Agent**モードを使用して、プロジェクト、クラス、またはメソッドの単体テストを生成できます。

##### チャットビューの3つのモード

| モード | 最適化されている用途 |
|--------|---------------------|
| **Ask** | コードプロジェクト、コーディングトピック、一般的なテクノロジの概念に関する質問 |
| **Edit（編集）** | コードベース内の複数のファイル間で編集を行う |
| **Agent（エージェント）** | エージェントコーディングワークフローを開始する |

> **重要**: エージェントモードでチャットビューを使用する場合、GitHub Copilotは1つのタスクを完了するために**複数のPremium要求**を行う場合があります。使用されるPremium要求の合計数は、タスクの複雑さ、関連するステップの数、選択したモデルによって異なります。

##### Ask モードを使用して単体テストを作成する

Ask モードは、ファイル内の複数の関数またはメソッドのテストを作成する場合や、ファイル全体のテストを作成する場合に便利です。

**手順:**

| ステップ | 操作 |
|---------|------|
| 1 | テストするコードを含むファイルを開く |
| 2 | チャットビューを開き、**Ask** モードを使用して新しいチャットセッションを開始 |
| 3 | チャットセッションにコンテキストを追加（下記参照） |
| 4 | ファイル内のコードの単体テストを求めるプロンプトを入力 |
| 5 | 推奨される単体テストを確認し、必要に応じてプロンプトを更新して結果を絞り込む |
| 6 | 推奨される単体テストをテストファイルに移動（**[エディターで適用]** ボタンも使用可能） |
| 7 | テストファイルを保存 |
| 8 | テストを実行して機能を確認 |
| 9 | 必要に応じてテストケースを追加・変更 |

**コンテキストの追加方法:**
- Visual Studio Codeのエクスプローラービューからチャットビューにファイルを**ドラッグ＆ドロップ**
- **[コンテキストの追加]** ボタンを使用
- `@workspace` チャット変数を使用してワークスペースを指定
- 外部ファイル（共同作成者のガイドラインなど）を開いてコンテキストに追加

**プロンプト例:**
```
@workspace /explain I need to create unit tests for the code in this file. The tests should be written in Python and use the unittest framework.
```

##### 編集モードを使用して単体テストを作成する

編集モードは、ファイル内の特定の関数またはメソッドのテストを作成する場合や、ファイル全体のテストを作成する場合に便利です。

**手順:**

| ステップ | 操作 |
|---------|------|
| 1 | テストするコードを含むファイルを開く |
| 2 | チャットビューを開き、**編集** モードを使用して新しいチャットセッションを開始 |
| 3 | チャットセッションにコンテキストを追加（下記参照） |
| 4 | 目的の単体テストを作成するためのプロンプトを入力 |
| 5 | 編集モードを使用して作成されたテストファイルを確認し、ファイルを保存または破棄 |
| 6 | 必要に応じて新しいプロンプトでファイルを更新 |
| 7 | テストファイルを保存 |
| 8 | テストを実行して機能を確認 |
| 9 | 必要に応じてテストケースを追加・変更 |

**編集モードでのコンテキスト追加:**
> **注意**: チャット参加者は編集モードでは使用できないため、`@workspace` は指定できません。

- `#codebase` を使用
- エクスプローラービューからチャットビューにファイルまたはフォルダーを**ドラッグ＆ドロップ**
- **[コンテキストの追加]** ボタンを使用して外部ファイルを追加

**プロンプト例:**
```
I need to create unit tests for the code in this file. The tests should be written in Python and use the unittest framework. Create a test file in the same directory as the code file.
```

##### エージェントモードを使用して単体テストを作成する

エージェントモードは、単体テストプロセス内のタスクを自動化できます。テストプロジェクトのスキャフォールディング、テストファイルの作成、テストの実行、テストレポートの生成などが可能です。**プロジェクトをより深く理解する必要がある単体テストを作成するのに最適**です。

**手順:**

| ステップ | 操作 |
|---------|------|
| 1 | テストするコードを含むファイルを開く |
| 2 | チャットビューを開き、**Agent** モードを使用して新しいチャットセッションを開始 |
| 3 | （オプション）[ツール]アイコンを選択して、使用するツールを構成 |
| 4 | 目的のタスクを定義するプロンプトを入力 |
| 5 | タスクを実行するエージェントモードの進行状況を監視 |
| 6 | ツールの呼び出しとターミナルコマンドを確認または拒否 |
| 7 | 作成・更新されたファイルを確認し、更新を保持または破棄 |
| 8 | 必要に応じて新しいプロンプトで特定のテストを修正・強化 |

**エージェントモードの特徴:**
- コンテキストを指定する必要がない（Copilotが**自動的に決定**）
- 編集する関連するコンテキストとファイルを自動的に検出
- **テストエクスプローラー** ツールでテストを実行
- **ターミナル** ツールでコマンドを実行
- 実行中のタスクを**中断**してコンテキストやツールを変更可能

**プロンプト例:**
```
Ensure that a suitable unit tests project is prepared for the selected code file. Create a test file in the unit test project that includes unit tests for all methods in the selected file. Unit tests should be written in C# and use the xUnit framework. Run the tests to ensure expected results.
```

> **ヒント**: エージェントモードが提案するツールの呼び出しとターミナルコマンドを確認または拒否できます。不要なコマンドは拒否することで、より効率的にテストを生成できます。

#### 2. インラインチャット
インラインチャットを使用すると、コードエディターで単体テストを直接作成できます。チャットビューに切り替えずに特定の関数またはメソッドのテストを作成しつつ、「テストの生成」スマートアクションよりも細かい制御が必要な場合に便利です。

**インラインチャットで単体テストを作成する手順:**

| ステップ | 操作 |
|---------|------|
| 1 | テストするコードを含むファイルを開く |
| 2 | テストするコードブロックを選択 |
| 3 | **Ctrl + I** キーボードショートカットでインラインチャットセッションを開く（またはGitHub Copilotメニューから **エディター インライン チャット** を選択）|
| 4 | プロンプトを入力（例: `/tests このメソッドの単体テストを生成します。成功と失敗の両方を検証し、エッジケースを含めます。`）|
| 5 | 推奨される単体テストを確認（テスト名の変更、テストケースの変更や削除、アサーションの追加など調整可能） |
| 6 | **[承諾]** または **[閉じる]** を選択して推奨される単体テストを受け入れるか破棄 |
| 7 | テストファイルを保存（通常は別の "tests" ディレクトリに保存） |
| 8 | プロジェクトをビルドしてテストファイルが含まれており実行できることを確認 |
| 9 | テストを実行して機能を確認 |
| 10 | 必要に応じてインラインチャットでテストを絞り込み、テストケースを追加または変更 |

> **ポイント**:
> - インラインチャットは **Ctrl + I** で開きます
> - ファイル全体または特定のコードの選択に対する単体テストを作成可能
> - 成功と失敗の両方のシナリオを検証し、エッジケースを含むテストを生成できます
> - 生成されたテストはコードエディターに表示され、既存のテストファイルでテストコードを生成するか、存在しない場合は新しいテストファイルが作成されます

#### 3. スマートアクション
**[テストの生成]**スマートアクションを使用して、プロンプトを記述せずに選択したコードの単体テストを生成します。

### テストの生成スマートアクションの詳細

テストの生成スマート アクションは、コードの単体テストを作成するためのGitHub Copilotの機能です。現在のファイル内のコードを分析し、コードの構造と動作に基づいてテスト ケースを生成します。この機能は、コードが徹底的にテストされ、品質基準を満たしていることを確認したい開発者に特に役立ちます。

#### ファイル全体の単体テストを生成する手順

ファイル内のすべての関数とメソッドのテストを個別に選択しなくても作成する場合に便利です。

| ステップ | 操作 |
|---------|------|
| 1 | テストするコードを含むファイルを開く |
| 2 | コードエディター内を右クリックし、**[Copilot]** を選択し、**[テストの生成]** を選択 |
| 3 | GitHub Copilotがコードを分析し、ファイル内のすべての関数とメソッドの単体テストを生成 |
| 4 | 生成されたテストを確認（テスト名の変更、テストケースの変更や削除、アサーションの追加など調整可能） |
| 5 | **[承諾]** または **[閉じる]** を選択して推奨される単体テストを受け入れるか破棄 |
| 6 | テストファイルを保存（通常は別の "tests" ディレクトリに保存） |
| 7 | テストを実行して機能を確認 |

> **ポイント**: 生成されるテストには、各関数またはメソッドのさまざまなシナリオに対応するアサーションとテスト ケースが含まれます。

#### 選択範囲の単体テストを生成する手順

ファイル全体のテストを生成せずに、特定の関数またはメソッドのテストを作成する場合に便利です。

| ステップ | 操作 |
|---------|------|
| 1 | テストするコードを含むファイルを開く |
| 2 | テストするコードブロックを選択 |
| 3 | 選択したコードを右クリックし、**[Copilot]** を選択し、**[テストの生成]** を選択 |
| 4 | GitHub Copilotが選択したコードを分析し、単体テストを生成 |
| 5 | 生成されたテストを確認し、必要な調整を行う |
| 6 | ファイルを保存 |
| 7 | テストを実行して機能を確認 |

> **ヒント**: 特定のメソッドのみをテストしたい場合は、選択範囲を使う方法が効率的です。

#### 4. コード行の入力候補
コード行の入力候補を使用して、既存のテストケースの単体テストの追加を提案します。

### ハンズオン演習

**演習1: チャットビューでテストを生成**
```
Write a unit test for the method in the #editor
```

**演習2: 選択したコードのテストを生成**
```
#selection write a unit test for this code
```

## レクチャー3: 【ハンズオン】エッジケースの特定とテスト提案
**時間目安: 12分**

### エッジケースと境界条件のテスト

GitHub Copilot Chatは、コードのコンテキストとセマンティクスを使用して、関数が正しく動作していることを確認するアサーションを提案します。

**Copilotが提案できるエッジケース:**
- エラー処理のテストケース
- null値のテストケース
- 予期しない入力の型のテストケース

### 単一メソッド vs 複数メソッドの選択

| シナリオ | 推奨プロンプト |
|---------|--------------|
| 単一メソッドがエディターで完全に表示されている場合 | `Write a unit test for the method in the #editor` |
| 複数のメソッドが表示されている場合、または目的のメソッドが表示範囲を超えて拡張されている場合 | コードを選択して `#selection write a unit test for this code` |

## レクチャー4: 既存テストの効果向上とボイラープレート生成
**時間目安: 10分**

### 失敗したテストの修正

Copilotは Visual Studio Codeのテストエクスプローラーと統合され、失敗したテストの修正に役立ちます。

**方法1: テストエクスプローラーから修正**
1. テストエクスプローラーで、失敗したテストにカーソルを合わせる
2. **[テストエラーの修正]**ボタン（スパークルアイコン）を選択
3. Copilotの推奨される修正プログラムを確認して適用

**方法2: スラッシュコマンドで修正**
1. チャットビューを開く
2. `/fixTestFailure` スラッシュコマンドを入力
3. Copilotの提案に従ってテストを修正

> **ヒント**: エージェントモードでは、テストの実行時にテスト出力が監視され、失敗したテストの修正と再実行が自動的に試行されます。

### Visual Studio Codeでの単体テスト環境

**必要なリソース:**
- .NET 8.0 SDK以降
- Visual Studio Code用のC#開発キット拡張機能
- プロジェクトに追加されたテストフレームワークパッケージ

### テストフレームワークの有効化

Visual Studio Codeのコマンドパレットから、各テストフレームワークを有効化できます。

| フレームワーク | コマンドパレットでの選択 | パッケージ参照 |
|--------------|----------------------|--------------|
| **xUnit** | [.NET: 新しいプロジェクト...] → [xUnit テスト プロジェクト] | `xUnit`, `xunit.runner.visualstudio` |
| **NUnit** | [.NET: 新しいプロジェクト...] → [NUnit3 テスト プロジェクト] | `NUnit`, `NUnit3TestAdapter` |
| **MSTest** | [.NET: 新しいプロジェクト...] → [MSTest テスト プロジェクト] | `MSTest.TestAdapter`, `MSTest.TestFramework` |

**テストプロジェクトへの参照追加:**
```bash
dotnet add [テストcsprojのパス] reference [テスト対象csprojのパス]
```

## レクチャー5: アサーション作成の支援
**時間目安: 10分**

### コンテキストとセマンティクスに基づくアサーション提案

GitHub Copilot Chatは、コードのコンテキストとセマンティクスを使用して、関数が正しく動作していることを確認するアサーションを提案します。

**アサーション提案の例:**
- 関数入力パラメーターが有効であることを保証するアサーション
- 戻り値が期待される型であることを確認するアサーション
- 境界条件でのアサーション

### 一貫性の維持（カスタム命令）

組織に特定のテスト要件がある場合は、カスタム命令を提供することで、Copilotがテストを生成する方法をカスタマイズできます。

**カスタマイズ可能な項目:**
- 推奨されるテストフレームワークを指定
- テストの名前付け規則を定義
- コード構造の基本設定を設定
- 特定のテストパターンまたは手法を要求

### Visual Studio Codeでのテスト管理

**C#開発キットの機能:**

| 機能 | 説明 |
|-----|------|
| **テストエクスプローラー** | ワークスペース内のすべてのテストケースを表示するツリービュー |
| **テストケースの実行/デバッグ** | クラスとメソッドの定義の左側にある緑色の再生ボタンから実行 |
| **テスト結果の表示** | エディター装飾とテストエクスプローラーの両方で状態が更新 |
| **テストコマンド** | コマンドパレットで「テスト:」を検索（「すべてのテストを実行する」など） |
| **テスト設定** | 設定エディターで「テスト」を検索 |

GitHub Copilotでテストフレームワークをセットアップする際に使用するコマンドは何ですか？

A) `/test`
B) `/setupTests`
C) `/createTest`

<summary>解答を見る</summary>

**正解: B) `/setupTests`**

解説: テストワークフローを高速化するために、チャットビューで `/setupTests` コマンドを入力することで、プロジェクトに適したテストフレームワークとVS Code拡張機能を設定できます。

### 問題2
GitHub Copilotで単体テストを生成する方法として**正しくないもの**はどれですか？

A) チャットビュー（Ask、Edit、Agentモード）
B) インラインチャット
C) ターミナルコマンド
D) スマートアクション（テストの生成）

<summary>解答を見る</summary>

**正解: C) ターミナルコマンド**

### 問題4
C#開発キットがサポートするテストフレームワークとして**正しくないもの**はどれですか？

A) xUnit
B) NUnit
C) MSTest
D) JUnit

<details>

### 問題31
単体テストにおける「Arrange、Act、Assert」パターンについて正しい説明はどれですか？

A) テストフレームワークの3つのモード
B) テストの構造を整理するパターンで、準備（Arrange）、実行（Act）、検証（Assert）の3つのセクションからなる
C) 3種類のテストフレームワーク
D) エラー処理の3つの段階

<details>
<summary>解答を見る</summary>

**正解: B) テストの構造を整理するパターンで、準備（Arrange）、実行（Act）、検証（Assert）の3つのセクションからなる**

解説: 「Arrange、Act、Assert」（AAA）パターンは、単体テストを構造化するための標準的なパターンです。Arrangeでテストに必要なオブジェクトや条件を準備し、Actでテスト対象のメソッドを実行し、Assertで結果が期待通りであることを検証します。

</details>
